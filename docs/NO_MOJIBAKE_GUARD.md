# NO_MOJIBAKE_GUARD — Обязательная защита от повреждения русского текста

## Обязательное правило перед любыми изменениями
Перед любыми будущими правками кода/контента/сборки/SEO разработчик или агент обязан сначала прочитать этот файл целиком. Без этого приступать к изменениям запрещено.

## Критические требования
1. Строго запрещено вносить mojibake/поврежденный русский текст в любой публичный контент сайта.
2. Публичный текст должен оставаться валидным, читаемым русским UTF-8.
3. Нельзя «исправлять» русский текст угадыванием или ручной перекодировкой, если текст уже читается нормально.
4. Нельзя вставлять в исходники текст из некорректно декодированного вывода терминала.
5. Нельзя сохранять файлы в ANSI/Windows-1251.
6. Нельзя использовать скрипты, которые читают/пишут HTML/JS/MJS/CSS/JSON/MD без явного сохранения UTF-8.
7. Если скрипт изменяет файлы, он обязан читать и записывать их в UTF-8.
8. Любое изменение, затрагивающее публичный текст, обязано включать проверку на mojibake до статуса PASS.
9. Проверять нужно не только исходники, но и результат сборки в dist.

## Обязательная проверка на подозрительные паттерны
Проверка должна искать типичные признаки mojibake, включая последовательности с:
- Р
- СЃ
- С‚
- СЂ
- Рќ
- Рџ
- Р°
- Рµ
- Рё
- Рґ
- РЅ
- Рѕ
- Р±
- Р»

Важно: нельзя падать только из-за одиночной буквы «Р», которая может встречаться в корректном тексте. Проверка должна анализировать именно подозрительные последовательности вида «Рќ», «Рџ», «СЃ», «С‚», «Р°» и т.п.

## Действия при обнаружении подозрительного mojibake
Если подозрительный mojibake найден в публичных исходниках или в dist, необходимо:
1. Немедленно остановиться.
2. Сообщить точный файл и строку/фрагмент.
3. Исправить проблему до продолжения работ.
4. Пересобрать проект.
5. Повторно запустить проверку.

## Исключения
1. Аудит-логи или исторические отчеты могут содержать старые поврежденные фрагменты, но это не должен быть публичный контент сайта.
2. Бинарные файлы нельзя сканировать как UTF-8-текст.

## Визуальная проверка публичных страниц
Если редактировались текст/контент, публичные страницы обязательно проверяются визуально. Задача не может считаться завершенной, если на публичных страницах присутствует mojibake.

## Standard verification command examples
Ниже безопасные примеры команд для Windows/PowerShell.

1. Поиск в исходниках (исключая служебные каталоги и при необходимости audit):
```powershell
Get-ChildItem -Path . -Recurse -File -Include *.html,*.js,*.mjs,*.css,*.json,*.md |
  Where-Object {
    $_.FullName -notmatch '\\node_modules\\' -and
    $_.FullName -notmatch '\\dist\\' -and
    $_.FullName -notmatch '\\audit\\'
  } |
  Select-String -Pattern 'Р[Ѐ-ӿ]|С[Ѐ-ӿ]|Рќ|Рџ|СЃ|С‚|Р°|Рµ|Рё' -AllMatches
```

2. Поиск в dist после сборки (HTML/JS/CSS):
```powershell
Get-ChildItem -Path .\dist -Recurse -File -Include *.html,*.js,*.css |
  Select-String -Pattern 'Р[Ѐ-ӿ]|С[Ѐ-ӿ]|Рќ|Рџ|СЃ|С‚|Р°|Рµ|Рё' -AllMatches
```

3. Примечание по исключениям:
- Всегда исключать бинарные/медиа-файлы (изображения, видео, архивы, шрифты и т.д.) из UTF-8 текстового сканирования.
